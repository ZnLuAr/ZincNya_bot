## 一个关键事实：category 不参与召回

这是所有分类决策的前提，理解它之后很多取舍就自然清楚了。

客户端采用的 BM25 评分公式（见 [llm-knowledge.md](llm-knowledge.md) 的「BM25 算法」节）只读取 `tags` / `title` / `content` / `priority`，完全不会把 `category` 纳入计算：

```
final = bm25(query, tags)*2.0 + bm25(query, title)*1.5 + bm25(query, content)*1.0 + priority*1.0
```

**`category` 不在公式里。** 它只承担两个次要作用：

1. 在控制台 `/llm knowledge list [category]` 筛选条目，方便批量查看、整理文件
2. 注入 LLM 上下文时作为前缀标注 `- [category] title: content`，给模型一个**阅读姿态提示**

由此能推出几条重要推论：

- 划分 category 的依据是 **「希望 LLM 用什么姿态读这组内容」**，不是单纯看主题是否相近
- 想调整条目被检索的难易程度，应该修改 `tags` 或 `priority`——改 `category` 对召回完全没用
- 内容放错分类不会导致召回失败，只会让上下文里的前缀提示语义错位，轻微干扰模型理解

## 现有的四个分类

本项目目前使用四个 category，围绕 LLM 角色的人格设计展开：

| category | 内容主题 | LLM 的阅读姿态 | 召回时机 | priority 惯例 |
|----------|----------|----------------|----------|---------------|
| `style` | bot **怎么说话** | 「按这个方式表达」 | 谈风格 / 进入特定情境（出错、被戳、开玩笑……） | 默认风格 = 1，情境风格 = 0 |
| `interests` | bot **知道什么、喜欢什么** | 「这是我的兴趣 / 知识」 | 用户聊到对应话题 | 0~1 |
| `pitfalls` | bot **不该怎么反应**（反例驱动） | 「识别并规避这种诱导」 | 检测到诱导 / 危险话题 / 容易翻车的场景 | 1 |
| `identity` | bot **是什么、不是什么**（本体事实） | 「这是我的自我认知，要内化」 | 用户质疑 / 讨论身份本体 | 1 |

### 为什么 `identity` 和 `pitfalls` 分开而不合并

两者主题相邻（都涉及「LLM 是谁」），但**阅读姿态相反**，这是不该合并的理由：

- `[identity] 身份边界：服务器与客户端` → LLM 读作**正面自我认知**，要内化成「我是什么」
- `[pitfalls] AI 身份认领陷阱` → LLM 读作**反例 / 警戒**，要识别并规避

合并成一类后，只能共用一个 `[category]` 前缀，这样就必然有一半内容的语义提示错位——要么让正面事实显得像「要避免的东西」，要么让反例显得像「事实」。而分开几乎没有代价（category 不影响召回），只在语义清晰度上有收益。

判据：

- 写 **「bot 是 / 不是什么」的正面本体陈述**，考虑使用 `identity`
- 写 **「被这样诱导 / 在这种场景下，不要这样反应」的反例应对** 时，则考虑使用 `pitfalls`

> 主题挨得近不构成同类，哪怕主题高度重合，只要阅读姿态一正一防，就不能合并。比如说，`identity-*` 与 `pitfalls-ai-identity-trap` 主题几乎重叠，但前者是「内化的自我认知」、后者是「需要被警惕的诱导」，阅读姿态相反，分属两类。**判据始终是姿态，不是主题。**

## 新增分类的门槛

新建 category 之前应该依次自问：

1. **能不能塞进现有四类之一？** 能就不要新建。先看「内容主题」那一列——新内容回答的是「怎么说话 / 知道什么 / 不该怎么反应 / 是什么」里的哪个？
2. **它回答的是一个全新维度的问题吗？** 不是上述四个维度的延伸，而是真正的新维度。
3. **预计会有 ≥ 2 条吗？** 单条内容不值得开新分类——挂到最贴近的现有类即可。
4. **新姿态确实和现有四类都不同吗？** 如果新内容的阅读姿态和某现有类相同，就归进去。

四条**都**满足才新建。新建时**必须同步更新本文档的分类表**，补一行说明它回答什么问题、什么阅读姿态、什么召回时机。

> 小提醒：宁可往现有分类多收纳内容，也不要轻易新增分类，稳定少的体系更好打理。分类的价值在于「少而稳定」。

## Frontmatter 字段规范

完整字段语义见 [llm-knowledge.md](llm-knowledge.md) 的「Markdown 文件格式」节。这里只讲**编写时的取舍**。

标准 md 头部模板参考：

```markdown
---
category: identity
title: 身份边界：服务器与客户端
tags: [身份, 服务器, 客户端, bot, 窗口]
tags_expanded: [...]        # 不手写，由 scripts/expand_knowledge_tags.py 生成
priority: 1
---
```

### category

四类之一，见上表。严格按照阅读姿态归类，不单纯只看素材主题。

### title

一句话浓缩「这条说的是什么」。参与 BM25（权重 1.5），所以**标题里出现的词等于召回关键词**。

习惯写法：`大类：具体主题`（如 `人类学：人本身与物质文化`、`反例：AI 身份认领陷阱`）。冒号前缀让同主题条目在 `list` 里聚拢，也给 LLM 一个语境锚点。

### tags（人工核心 tags）

**这是召回的命脉**（权重 2.0，最高）。原则：

- **写「区分性」的词，不写「概括性」的词。** `tags` 是用来让对的查询命中、错的查询落空的。像 `人格`、`对话`、`AI`、`交流`、`沟通`、`tone`、`personality` 这类过于宽泛的词会让大量无关消息误命中，**要避免**。
- 4~6 个核心词即可。语义泛化（同义词、相关词、中英对照）交给 `tags_expanded` 自动扩展，不用人工堆。
- 想象用户会用什么词触发这条，再把那些词写下去。
- 中英混排没问题，tokenizer 同时处理（见 [llm-knowledge.md](llm-knowledge.md) 的「分词规则」节）。

> 据野史记载，ZincPhos 一度在一篇知识库条目的 `tags_expanded` 中塞了 60+ 个标签，塞满 `沟通 / communication / interaction / persona / tone` 这类宽泛词，导致几乎任何闲聊都召回它，这导致了大量的上下文占用，不仅浪费了资源，还稀释了模型的注意力、降低生成质量。应该注意：**tags 并非多多益善，膨胀只会招致更大的噪声**。

### tags_expanded

**可以不用手写。** 修改人工 tags 后运行 `scripts/expand_knowledge_tags.py` 离线调 LLM 生成（见 [llm-knowledge.md](llm-knowledge.md) 的「标签扩展工作流」节）。

如果发现某条 `tags_expanded` 膨胀或塞了宽泛词，只需要精简人工 tags 再重跑脚本，直接手改会被下次运行覆盖，还会打乱文件哈希判断逻辑。

### priority

BM25 分数的固定加成（`+= priority * 1.0`）。惯例：

- `1`：默认就该常驻 / 高频召回的（默认风格、身份本体、关键反例）
- `0`：按话题召回就好、不需要额外加权的（绝大多数 `interests` 条目、情境化 `style`）
- `≥ 2`：仅在「这条无论如何都要压过其他条目」时才用，慎用——会挤掉真正更相关的条目

## 正文写法约定

这些不是硬性规则，但按这个写法能或多或少提升素材对模型的参考价值：

- **正反示例并列**：用 `✅` / `❌` 给出「该这样 / 不该这样」的对照，比纯描述更能校准行为。`pitfalls-*` 和 `style-*` 大量用这个格式。
- **示例用 bot 的真实口吻**，而不是干巴巴罗列抽象规则——LLM 会模仿示例的语气。
- **一条只讲一个概念**（one file one concept）。概念变大就拆文件——拆开后召回更精准，也不会一次塞一大坨进上下文。
- **避免跨条目重复**：同一规则别在多个文件里各写一遍。重复内容会在召回时占用多余的 token，后续修改还容易出现内容不一致。

## 改完之后

编辑任何 `.md` 后，要使让改动生效，应有：

1. 新增 / 改动标题、人工 tags 时，可以运行标签扩展脚本：python scripts/expand_knowledge_tags.py
2. 重建知识库索引：在客户端 CLI 中 `/llm knowledge reindex`，或重启 bot 以自动重建
3. 推荐用 `/llm knowledge search <query>` 校验（`utils/command/llm.py`），模拟提问测试，确认预期问句能正常命中、打分合理。

> Windows 控制台上，如果运行脚本时遇到 `UnicodeEncodeError`（GBK 编码 emoji），可以加上环境变量 `PYTHONIOENCODING=utf-8`。

## 维护原则速记

1. **先问「能塞进现有四类吗」**，能就不新建分类
2. **category 选「希望 LLM 用什么姿态读」**，不是「主题像不像」
3. **tags 写区分性的词**，宽泛词是召回噪声的来源
4. **tags_expanded 不手写**，要改去改人工 tags 再重跑脚本
5. **一条只讲一个概念**，概念变大就拆文件
6. **新建 category 必须同步更新本文档的分类表**
